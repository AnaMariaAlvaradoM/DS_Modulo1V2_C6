¿Qué es persistencia?

Definición técnica

La persistencia es la capacidad de que un dato sobreviva al cierre del programa que lo creó. Un dato persistente sigue existiendo aunque apagues el computador.

🌎 En la vida real: escribir en un cuaderno (persistente) vs. recordar algo solo de memoria (volátil). Si te golpeas la cabeza, pierdes la memoria; el cuaderno sigue ahí. La RAM es la memoria; la base de datos es el cuaderno.

¿Por qué un CSV no alcanza?

🔍

No sabe buscar

Para filtrar tienes que cargar todo el archivo a memoria. Una base busca al instante.

👥

No controla accesos

¿Qué pasa si dos procesos escriben al mismo tiempo? El CSV se corrompe.

🛡️

No garantiza integridad

Puedes escribir basura sin que nadie reclame. La base valida tipos y reglas.

📈

No escala

Un CSV nuevo cada día es inmanejable. ¿Cómo cuentas las cancelaciones de marzo?

● Herramienta · SQL

🔧SQLAlchemy y el ORM

Antes de tocar la base, entendamos con qué le vamos a hablar. Escribir SQL crudo pegando strings dentro de Python es peligroso. Aquí entra el ORM.

El problema de escribir SQL a mano

Para hablar con una base SQL, podríamos armar el comando como un texto gigante, concatenando variables a mano: "INSERT INTO predicciones VALUES (" + valor + ")". Eso trae tres riesgos serios:

💉

Inyección SQL

Una vulnerabilidad de seguridad clásica: alguien mete código malicioso por una variable.

🐛

Errores tardíos

Los fallos de sintaxis solo aparecen al ejecutar, no al escribir.

🔄

Amarrado al motor

Si cambias de SQLite a PostgreSQL, reescribes todo el SQL.

¿Qué es un ORM?

Definición técnica

ORM = Object-Relational Mapping (Mapeo Objeto-Relacional). Es una herramienta que traduce entre objetos de Python (clases) y tablas de la base de datos. Trabajas con clases y objetos; el ORM genera el SQL por ti.

🌎 En la vida real: un traductor en una reunión. Tú hablas español (Python), el otro habla inglés (SQL). No aprendes inglés; el traductor convierte en ambas direcciones. SQLAlchemy es ese traductor.

¿Por qué SQLAlchemy?

Es el estándar de la industria en Python para ORM. Funciona con SQLite, PostgreSQL, MySQL y más, con el mismo código. Hoy usamos SQLite por debajo (no requiere instalar un servidor), pero el código sirve igual en producción con PostgreSQL: solo cambia una línea de conexión.

terminal · entorno master-ia 
pip install sqlalchemy
⚠️
Cuidado con el nombre: es sqlalchemy (s-q-l), no "sqalchemy". Un typo da el error No matching distribution found. La versión esperada es 2.0.x (usamos la sintaxis moderna 2.0).

📐El plano de la base

Primer archivo: modelos_db.py. Es el plano de la base de datos. Define cómo es la tabla y cómo conectarse. Todos los demás archivos lo importarán.

📁

¿Qué necesitamos guardar de cada predicción?

Para poder auditar después, cada predicción necesita: el id del pedido, la categoría, el valor, la probabilidad que dio el modelo, la predicción final (sí/no), la versión del modelo y la fecha. Sin versión ni fecha, no hay auditoría posible. De esa lista nacen las columnas.

Parte A — Las importaciones

modelos_db.py · inicio 
from datetime import datetime
from sqlalchemy import String, Integer, Float, Boolean, DateTime, create_engine
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column
Qué hace cada pieza

create_engine crea la conexión · DeclarativeBase es la clase raíz de los modelos · Mapped y mapped_column son la forma moderna (2.0) de declarar columnas.

Parte B — La tabla como clase Python

modelos_db.py · a continuación 
class Base(DeclarativeBase):
    pass


class PrediccionCancelacion(Base):
    __tablename__ = "predicciones"

    id: Mapped[int] = mapped_column(Integer, primary_key=True, autoincrement=True)
    id_pedido: Mapped[str] = mapped_column(String(20))
    categoria: Mapped[str] = mapped_column(String(50))
    valor_pedido: Mapped[float] = mapped_column(Float)
    prob_cancelacion: Mapped[float] = mapped_column(Float)
    prediccion: Mapped[bool] = mapped_column(Boolean)      # True = se cancelará
    modelo_version: Mapped[str] = mapped_column(String(20))
    creado_en: Mapped[datetime] = mapped_column(DateTime)
¿Y esto qué me dio?

Una tabla definida sin escribir una sola línea de SQL. La clase PrediccionCancelacion ES la tabla predicciones. Cada objeto que crees de esta clase será una fila.

🔑
primary_key=True marca la columna que identifica de forma única cada fila (como la cédula). autoincrement=True hace que la base asigne el id sola: 1, 2, 3…

Antes de seguir con código: ¿quién creó el modelo que predice cancelaciones? ¿De dónde salen esas predicciones que vamos a guardar? En una empresa real, esto está dividido en roles.

ROL 2
🚀

ROL 3
🛠️

No entrena el modelo. Recibe sus predicciones y las persiste de forma ordenada y auditable.

¿Por qué simulamos con números aleatorios?

En la práctica no tenemos el modelo real de Rappi conectado. Entonces lo simulamos: en vez de llamar al modelo, generamos predicciones de mentira con números aleatorios, para tener datos realistas que persistir. En una empresa, esa parte sería una llamada al modelo real (modelo.predict(pedido)); aquí la reemplazamos por random solo para aprender.

¿Y la semilla (seed)?

Para qué sirve

Cuando generas números aleatorios, cada computador daría números distintos. random.seed(42) fija el punto de partida del generador: todos obtienen exactamente los mismos números. Así los resultados son reproducibles y verificables.

💡 La semilla no existe en producción: ahí los datos son reales, no generados. Es solo una herramienta de aprendizaje. El número 42 es una convención popular, sin significado técnico.

?Si el modelo ya está entrenado y corriendo en otro lado, ¿por qué nuestro código genera predicciones con random en vez de usar el modelo real?
▶ Ver respuesta
Porque en el ejercicio no tenemos el modelo de producción conectado. Lo simulamos para tener datos realistas que guardar y auditar. En la empresa, random.uniform(0,1) sería reemplazado por la salida real del modelo. Lo que aprendemos —persistir y auditar— es idéntico en ambos casos.
● Archivos 2, 3 y 4

💾Guardar, auditar y comprobar

El corazón de la clase. Generamos 500 predicciones, las guardamos en la base, las consultamos para auditar, y comprobamos que sobreviven al cierre del programa.

Generar las predicciones

guardar_predicciones.py 
import random
from datetime import datetime
from sqlalchemy.orm import Session
from modelos_db import engine, PrediccionCancelacion

random.seed(42)

categorias = ["Chicken", "Beef", "Side", "Pasta", "Seafood", "Dessert", "Vegetarian"]

predicciones = []
for i in range(500):
    prob = round(random.uniform(0, 1), 3)
    predicciones.append({
        "id_pedido": f"PED{1000 + i}",
        "categoria": random.choice(categorias),
        "valor_pedido": round(random.uniform(15000, 95000), 0),
        "prob_cancelacion": prob,
        "prediccion": prob >= 0.5,
        "modelo_version": "v1.0",
    })

print("Predicciones generadas:", len(predicciones))
📌
La línea from modelos_db import engine, PrediccionCancelacion es la que conecta los archivos: trae la conexión y la tabla desde el plano. No se redefinen aquí, se reutilizan.

Guardar con Session

guardar_predicciones.py · al final 
ahora = datetime.now()

with Session(engine) as session:
    objetos = [PrediccionCancelacion(creado_en=ahora, **p) for p in predicciones]
    session.add_all(objetos)
    session.commit()
    print("Predicciones guardadas y confirmadas:", len(objetos))
▸ salida en consola
[modelos_db.py] Tabla lista en rappi_produccion.db
Predicciones generadas: 500
Predicciones guardadas y confirmadas: 500
¿Cómo se lee?

Session(engine) abre la sesión de trabajo · **p desempaqueta cada diccionario en argumentos · add_all agrega los 500 objetos · commit() los escribe de verdad en la base.

⚠️
Error silencioso más común: olvidar session.commit(). No sale ningún error, el programa "corre bien"… pero la base queda vacía. Sin commit, nada se guarda.

Auditar lo guardado

Auditar es interrogar lo que el modelo predijo. Lo mejor: no necesitas volver a correr el modelo, solo consultas la base.

auditoria.py · solo lectura 
from sqlalchemy import select, func
from sqlalchemy.orm import Session
from modelos_db import engine, PrediccionCancelacion

with Session(engine) as session:
    total = session.scalar(select(func.count()).select_from(PrediccionCancelacion))
    print("Total de predicciones guardadas:", total)

    alto = session.scalars(
        select(PrediccionCancelacion).where(PrediccionCancelacion.prob_cancelacion >= 0.8)
    ).all()
    print("Pedidos de alto riesgo (prob >= 0.8):", len(alto))

    prom = session.scalar(select(func.avg(PrediccionCancelacion.prob_cancelacion)))
    print("Probabilidad promedio de cancelación:", round(prom, 4))
▸ salida en consola
Total de predicciones guardadas: 500
Pedidos de alto riesgo (prob >= 0.8): 92
Probabilidad promedio de cancelación: 0.4991

Cancelaciones por categoría

auditoria.py · agrupar y ordenar 
with Session(engine) as session:
    consulta = (
        select(PrediccionCancelacion.categoria, func.count())
        .where(PrediccionCancelacion.prediccion == True)
        .group_by(PrediccionCancelacion.categoria)
        .order_by(func.count().desc())
    )
    print("Cancelaciones predichas por categoría:")
    for categoria, cantidad in session.execute(consulta).all():
        print(f"  {categoria}: {cantidad}")

● Archivo 5 · MongoDB

🍃NoSQL con MongoDB

Rappi ya no tiene un solo modelo: tiene versiones (v1.0 RandomForest, v2.0 XGBoost…) y cada una con campos distintos. Para guardar esos metadatos irregulares, SQL estorba y NoSQL brilla.

El problema que resuelve NoSQL

Si guardas los metadatos de cada modelo en una tabla SQL rígida, cuando v2.0 trae un campo nuevo (hiperparametros) que v1.0 no tenía, toca alterar la tabla y las filas viejas quedan con esa columna vacía. La estructura fija estorba cuando los datos son irregulares.

Definición técnica

Una base NoSQL documental guarda documentos (estructuras tipo JSON) en lugar de filas. Cada documento puede tener campos distintos. No hay esquema rígido previo.

🌎 En la vida real: SQL es un formulario con casillas fijas, todos llenan los mismos campos. MongoDB es una hoja en blanco, cada quien escribe lo que su caso necesita. Para datos uniformes, el formulario; para datos variables, la hoja en blanco.

¿Por qué Atlas y no instalar MongoDB?

MongoDB Atlas es MongoDB en la nube, gratis en su capa M0. Evita instalar un servidor en cada máquina. Te conectas con una URI y listo. La preparación (cuenta, cluster M0, usuario, URI) se hace una sola vez.

modelo_v2 = {
    "nombre": "predictor_cancelaciones",
    "version": "v2.0",
    "algoritmo": "XGBoost",
    "metricas": {"accuracy": 0.91, "precision": 0.88, "recall": 0.85, "f1": 0.86},

¿Y esto qué me dio?

Dos modelos con estructuras diferentes conviviendo en la misma colección. notas es texto libre; hiperparametros solo existe en v2.0. En SQL esto habría exigido alterar la tabla. Aquí, cero fricción.

READ — leer y filtrar (incluso campos anidados)

metadatos_modelos.py 
total = coleccion.count_documents({})
print("Documentos en la colección:", total)

v2 = coleccion.find_one({"version": "v2.0"})
print("v2.0 accuracy:", v2["metricas"]["accuracy"])

buenos = coleccion.count_documents({"metricas.accuracy": {"$gte": 0.90}})
print("Modelos con accuracy >= 0.90:", buenos)
▸ salida en consola
Documentos en la colección: 2
v2.0 accuracy: 0.91
Modelos con accuracy >= 0.90: 1
🔎
La notación de punto "metricas.accuracy" entra a campos anidados. $gte significa "mayor o igual" (greater than or equal).

UPDATE y DELETE

metadatos_modelos.py 
# UPDATE: marcar v2.0 como modelo en producción
coleccion.update_one(
    {"version": "v2.0"},
    {"$set": {"estado": "produccion"}}
)

# DELETE: dar de baja el modelo viejo v1.0
coleccion.delete_one({"version": "v1.0"})
print("Documentos tras eliminar v1.0:", coleccion.count_documents({}))
▸ salida en consola
Documentos tras eliminar v1.0: 1
¿Cómo se lee?

$set agrega o cambia un campo sin tocar el resto: a v2.0 le aparece estado, pero v1.0 nunca lo tuvo. Campos distintos por documento, sin problema.

?Rappi quiere guardar, por cada pedido, los comentarios libres que el cliente escribe al cancelar ("llegó frío", "demoró mucho", emojis…). ¿Eso va en SQL o en NoSQL?
▶ Ver el razonamiento
No hay una sola respuesta correcta. Si es texto libre, variable, a veces vacío y a veces con metadatos extra, encaja en NoSQL. Pero si es solo un texto corto atado a un id_pedido que ya está en la tabla de predicciones, podría ser una columna más en SQL y mantener todo cruzable. La clave es elegir según qué tan estructurado y cruzable necesitas el dato. Eso es ingeniería de datos real.
