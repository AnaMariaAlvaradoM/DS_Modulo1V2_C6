El problema de negocio

El gerente pide una decisión con datos

Todo arranca con una decisión real, no con un archivo.

Rappi nació en Bogotá en 2015 y hoy mueve millones de pedidos al día. Eres parte del equipo de datos. El gerente de producto dice: "Quiero lanzar una categoría de comida nueva en Colombia. No me digan cuál creen ustedes; díganme cuál, con datos."

🤔Con SOLO el archivo de pedidos de Rappi Colombia, ¿puedes decirle al gerente qué categoría nueva lanzar? ¿Qué te falta?
▸ Ver la idea clave
El archivo solo dice lo que ya se pide; no dice lo que existe en el mundo y aún no se ofrece. Mirar lo más vendido y lanzar más de eso es repetir lo que ya funciona, no lanzar algo nuevo. Para encontrar algo nuevo necesitas saber qué se cocina en el mundo — y eso vive afuera, en una API.
🌉
La idea que abre la clase: tu archivo solo conoce a Colombia. Para algo nuevo necesitas dos cosas: qué se cocina en el mundo (una API) y hacia dónde se inclina Colombia (los pedidos). La tubería que trae, limpia y cruza las dos fuentes se llama pipeline ETL.
🧩
Concepto

Pipeline ETL: la idea nueva

La tubería automática de datos
🤔Una planta de agua: entra agua sucia del río, pasa por filtros, sale agua potable. Tres etapas fijas, en orden. ¿Qué tres etapas tendría una "planta de datos"?
▸ Ver la respuesta
Entrada, limpieza y salida. Eso es exactamente ETL: Extraer → Transformar → Cargar.
¿Qué es un pipeline?

"Pipeline" significa "tubería". Un pipeline ETL es una tubería automática de datos con tres etapas fijas: traer, limpiar, guardar. No es una tarea manual.

El flujo ETL de la clase
Extract → Transform → Load
Extract
Traer datos crudos de su fuente. Aquí usamos DOS fuentes: una API y un archivo.
Recoger agua de dos ríos distintos.
Transform
Limpiar, dar forma y cruzar las fuentes.
El filtro que vuelve potable el agua.
Load
Guardar el dato limpio. En esta clase lo dejamos listo; el guardado real es la Clase 2.
Embotellar el agua ya potable.
🧠
¿Por qué esto es EL trabajo del científico de datos? Un modelo de IA aprende de los datos que le das. Si entran datos sucios, aprende mal: "garbage in, garbage out". Por eso se invierte la mayor parte del tiempo en ETL: construir la cocina que le sirve comida limpia al modelo.

Concepto

¿Qué es una API?

Desde cero, con la analogía del restaurante
🤔En un restaurante no entras a la cocina a cocinar. Le pides al mesero, él lleva el pedido y trae el plato. ¿Por qué no te dejan entrar directo a la cocina?
▸ Ver la idea clave
Sería un caos, es un espacio privado, no sabrías dónde está todo. Esa es justo la razón de existir de una API: un intermediario con reglas que protege "la cocina" interna.
¿Qué es una API?

Es un mesero entre programas: pides un dato con ciertas reglas y te lo trae, sin dejarte entrar a "la cocina" (el sistema interno de otra empresa). API viene de Application Programming Interface.

⚓
Para recordar: API = el mesero. Pides con reglas, te trae el dato. No entras a la cocina ajena.
Endpoint
La dirección exacta a la que pides; es una URL.
La ventanilla donde haces el pedido.
Request
La petición que envías. En Python: requests.get(endpoint).
Decirle al mesero "quiero esto".
Status code
Número que dice si salió bien: 200 = bien · 404 = no existe · 403 = sin permiso.
"Listo" o "no hay".
JSON
El formato del dato que llega: texto con llaves y valores, casi idéntico a un diccionario de Python.
El plato servido en formato estándar.

6. Empaca todo en la función, con red de seguridad try/except, y quédate solo con la columna que importa.

python clase1_etl.ipynb
def extract_menu_mundial():
    url = "https://www.themealdb.com/api/json/v1/1/categories.php"
    try:
        datos = requests.get(url, timeout=10).json()
        return pd.DataFrame(datos["categories"])[["strCategory"]]
    except Exception:
        print("API no disponible")
        return pd.DataFrame()
🛡️
Buena práctica: el try/except evita que el pipeline muera si la API se cae; devuelve tabla vacía y el programa sigue en vez de explotar.

7. Úsala de verdad. Definir fue escribir la receta; esta línea es cocinarla.

🔁
Definir ≠ ejecutar: def crea la herramienta pero no la usa. Es la diferencia entre tener un exprimidor y exprimir la naranja.

Concepto

Archivos masivos: cuando no cabe en la RAM

Leer por lotes con chunksize
🤔Tu computador tiene 8 GB de RAM. Rappi te manda un archivo de pedidos de 20 GB. Si haces pd.read_csv("archivo.csv") como siempre… ¿qué pasa?
▸ Ver la respuesta
Pandas intenta meter 20 GB en 8 GB y se queda sin memoria: se traba.
¿Qué es la RAM?

La memoria de trabajo del computador: rápida pero limitada. Pandas mete ahí el archivo para procesarlo. Si el archivo es más grande que la RAM, no cabe.

🍕
Idea central: un archivo masivo es uno tan grande que no cabe entero en la RAM. Solución: leerlo por lotes (chunks), un pedazo a la vez. Como comerse una pizza por tajadas, no de un solo bocado.
chunksize=N

Le dice a read_csv: "tráeme de a N filas". Procesas un lote, lo sueltas, traes el siguiente. La RAM nunca se llena.

🔮
El límite de esta técnica: chunksize es la primera defensa, en UN solo computador. Cuando ni por lotes alcanza, se reparte el trabajo entre muchos computadores: eso es Big Data con PySpark, Unidad 2 de este mismo módulo.
🕳️

    return df

Transform · limpiar porque el conteo miente

Resultado: la función transform()

Aquí no se limpia "porque toca". Se limpia porque al contar algo sale un número absurdo, y limpiar es lo único que lo arregla.

"¿Cuántos pedidos se cancelaron?"

python clase1_etl.ipynb
(pedidos["estado"] == "cancelado").sum()
Resultado — sospechosamente bajo
34  ·  ¿solo 34 cancelados en 1207 pedidos? Huele raro.

Miremos todos los valores distintos de estado:

python clase1_etl.ipynb
sorted(pedidos["estado"].unique())
Resultado — la mentira al desnudo
[' cancelado', ' en_camino', ' entregado', 'CANCELADO', 'Cancelado', 'ENTREGADO', 'EN_CAMINO', 'En_camino', 'Entregado', 'cancelado', 'en_camino', 'entregado']
🎭
Son 3 estados disfrazados de 12: mayúsculas, minúsculas y espacios sobrantes. El conteo de "34 cancelados" solo agarró los que decían exactamente cancelado en minúscula sin espacios.
🤔¿Se limpia el texto por estética, o porque la respuesta al gerente está mal hasta que lo hagas?
▸ Ver la respuesta
Está mal. La limpieza no es cosmética; es la diferencia entre decir "34 cancelados" (falso) o el número real.

Normaliza: quita espacios y pasa a minúscula. Las 12 variantes colapsan a 3. Cuenta de nuevo, ahora bien:

python clase1_etl.ipynb
pedidos["estado"] = pedidos["estado"].astype(str).str.strip().str.lower()
(pedidos["estado"] == "cancelado").sum()
Resultado — la verdad
117  ·  el número real era 117, no 34. Limpiar más que triplicó la respuesta.
🏙️
El mismo problema esconde la ciudad: BOGOTA, Bogotá, bogota parecen tres ciudades pero son una. Se normaliza igual + quitar tildes → quedan 5 ciudades reales: Barranquilla, Bogota, Cali, Cartagena, Medellin.

Los nulos también mienten en los promedios

minutos_entrega tiene nulos, negativos e imposibles (999). Un solo 999 dispara el promedio.

python clase1_etl.ipynb
import numpy as np
pedidos["minutos_entrega"] = pd.to_numeric(pedidos["minutos_entrega"], errors="coerce")
pedidos.loc[(pedidos["minutos_entrega"] <= 0) | (pedidos["minutos_entrega"] > 180),
            "minutos_entrega"] = np.nan
pedidos["minutos_entrega"] = pedidos["minutos_entrega"].fillna(pedidos["minutos_entrega"].median())
🤔¿Por qué se rellena con la mediana y no con el promedio?
▸ Ver la respuesta
La mediana no se deja arrastrar por un valor extremo; el promedio sí. Un solo 999 envenena el promedio pero casi no mueve la mediana.

Y la cura del cruce anterior: .str.strip() quita el espacio invisible, así ' P0202' vuelve a ser 'P0202' y los 96 perdidos cruzan. Toda la limpieza empacada en su función:

python clase1_etl.ipynb · función transform completa
def transform(pedidos, catalogo):
    pedidos["estado"] = pedidos["estado"].astype(str).str.strip().str.lower()
    pedidos["ciudad"] = (pedidos["ciudad"].astype(str).str.strip().str.lower()
                         .str.normalize("NFKD")
                         .str.encode("ascii", "ignore").str.decode("utf-8")
                         .str.capitalize())
    pedidos["propina_usd"] = pd.to_numeric(pedidos["propina_usd"], errors="coerce").fillna(0)
    pedidos["minutos_entrega"] = pd.to_numeric(pedidos["minutos_entrega"], errors="coerce")
    pedidos.loc[(pedidos["minutos_entrega"] <= 0) | (pedidos["minutos_entrega"] > 180),
                "minutos_entrega"] = np.nan
    pedidos["minutos_entrega"] = pedidos["minutos_entrega"].fillna(pedidos["minutos_entrega"].median())
    pedidos = pedidos.dropna(subset=["id_pedido"]).drop_duplicates()
    pedidos["id_plato"] = pedidos["id_plato"].astype(str).str.strip()
    final = pedidos.merge(catalogo, on="id_plato", how="left")
    final["categoria"] = final["categoria"].fillna("Desconocido")
    return final
🔥
Cada línea está aquí por una razón: sin ella, una pregunta de negocio se respondía mal. La limpieza ahora vive dentro del pipeline y limpia mientras el dato fluye hacia el análisis.

Concepto + código

Validación de esquema: el contrato

Que el pipeline avise en vez de entregar basura
🤔En un contrato de arriendo hay condiciones obligatorias: precio, fecha, firmas. Si falta una, no vale. ¿Qué condiciones obligatorias debería cumplir la tabla final?
▸ Ver la respuesta
Estados sin variantes raras, minutos en un rango lógico, categoría siempre asignada. Esas condiciones son el esquema: columnas correctas y valores permitidos.
¿Qué hace assert?

Revisa una condición: si es verdadera, sigue; si es falsa, DETIENE el programa con el mensaje dado. El pipeline se frena solo si algo está mal.

python clase1_etl.ipynb
def validar_esquema(df):
    assert df["estado"].isin(["entregado", "cancelado", "en_camino"]).all(), "Estado inválido"
    assert df["minutos_entrega"].between(1, 180).all(), "Minutos fuera de rango"
    assert df["categoria"].notna().all(), "Categoría sin asignar"
    print("Esquema válido:", df.shape[0], "filas listas para cargar")
    return True
🤔¿Por qué assert y no un print de advertencia?
▸ Ver la respuesta
assert detiene el pipeline si algo falla. Un print se ignora y los datos malos siguen de largo. El assert es el guardia que no deja pasar basura.

def run_etl(ruta):
    menu = extract_menu_mundial()
    pedidos = extract_pedidos(ruta)
    catalogo = pd.read_csv("catalogo_platos.csv")
    final = transform(pedidos, catalogo)
    top = (final["categoria"].value_counts(normalize=True) * 100).round(1).head(3)
    print("Colombia pide (top 3):"); print(top)
    ausentes = set(menu["strCategory"]) - set(final["categoria"])
    print("Ausentes (candidatas):", sorted(ausentes))
    validar_esquema(final)
    return final
